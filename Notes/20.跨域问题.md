# 跨域问题

跨域问题，本质是**浏览器的同源策略**对前端发起的网络请求做出的安全限制：当一个页面的源，和它请求的资源/接口的源不一致时，浏览器会阻止这次请求的响应结果被页面获取（请求本身可能已发送到服务器，但前端拿不到数据），以此防止恶意网站窃取其他网站的用户数据（如cookie、token、会话信息）。

**同源：协议、端口、域名 三者完全一致**。


## 跨域问题的关键点
1. **仅存在于浏览器端**：服务器与服务器之间的请求 **没有跨域限制**，因为同源策略是浏览器为了保护用户本地数据设计的，非浏览器环境不生效。
2. **请求未被拦截，响应被拦截**：跨域时，前端的请求其实已经发送到目标服务器，服务器也会返回响应，但浏览器收到响应后，发现违反同源策略，会在控制台抛出跨域错误，并拒绝将响应数据交给前端JS。
3. **并非所有的请求都跨域**：浏览器对脚本发起的网络请求（AJAX、Fetch、axios）做跨域限制，而静态资源加载默认允许跨域（通过cors/cdn的跨域配置实现）


## 最常见的跨域场景

前端开发中**本地开发环境**是跨域的高频场景：
- 前端本地服务：`http://localhost:3000`（Vue/React/Angular 的开发服务器）
- 后端接口服务：`http://localhost:8080`（SpringBoot/Go/Node.js 的接口服务器）
此时前端请求后端接口，端口不同，浏览器会直接抛出跨域错误，这也是新手开发最容易遇到的跨域问题。


## 跨域不等于CORS

- **跨域：**是问题现象（请求被浏览器阻止）
- **CORS（跨域资源共享）：**是W3C标准的解决方案（通过服务器设置 HTTP 响应头，告诉浏览器允许哪些源的请求访问），也是目前前后端分离架构中最推荐、最通用的跨域解决方式。

除此之外，还有代理转发（开发环境用）、JSONP（老旧方案，仅支持 GET 请求）、WebSocket（无跨域限制，适用于实时通信）等解决方案。


## Golang Gin CORS中间件
`go get github.com/gin-contrib/cors`

**使用：**

```go
package main

import (
    "github.com/gin-gonic/gin"
    "github.com/gin-contrib/cors"
    "time"
)

func main() {
    r := gin.Default()
    r.Use(cors.New(cors.Config{
        AllowOrigins:     []string{"http://localhost:3000"},
        AllowMethods:     []string{"GET", "POST", "PUT", "DELETE"},
        AllowHeaders:     []string{"Origin", "Content-Type"},
        ExposeHeaders:    []string{"Content-Length"},
        AllowCredentials: true,
        MaxAge: 12 * time.Hour,
    }))
    r.GET("/ping", func(c *gin.Context) {
        c.JSON(200, gin.H{"message": "pong"})
    })
    r.Run(":8080")
}
```
- AllowOrigins：允许跨域的源（如 ["http://localhost:3000"]）
- AllowMethods：允许的方法（如 ["GET", "POST"]）
- AllowHeaders：允许的请求头
- ExposeHeaders：暴露给前端的响应头
- AllowCredentials：是否允许携带 cookie
- MaxAge：预检请求的缓存时间

**常见问题：**
- 如果 AllowOrigins 设置为 *，AllowCredentials 必须为 false，否则会报错。
- 建议根据实际前端地址配置 AllowOrigins，避免安全隐患。
