# WebSocket

## 简介
随着Web应用的复杂化和实时性需求的增长，HTTP协议的局限性逐渐暴露出来。WebSocket协议应运而生，为实时双向通信提供了完美的解决方案。

**HTTP协议的局限性**
1. 请求-响应模式的限制
   - 单向通信：客户端必须主动发起请求，服务端才能响应
   - 无法主动推送：服务器无法主动向客户端推送数据
   - **轮询开销**：要实现 “实时” 效果，客户端必须不断轮询服务器
2. 连接开销
   - 每次HTTP请求都需三次握手、携带完整的HTTP头部、连接管理（短连接需要频繁建立和关闭，长连接虽然可以复用，但仍然受限于请求-响应模式）
3. 实时性不足
   - 聊天应用：用户发送消息后，其它用户需要实时收到
   - 在线游戏：游戏状态需要实时同步
   - 股票行情：价格变化需要立即推送
   - 协作编辑：多人编辑文档时，需要实时看到对方的修改

**WebSocket设计目标**
- 全双工通信：客户端和服务器可以同时发送和接收数据
- 低延迟：建立连接后，数据传输延迟极低
- 低开销：数据帧头部仅2-14字节，远小于HTTP头部
- 持久连接：一次握手，长期使用
- 跨域支持：原生支持跨域通信


## WebSocket和HTTP对比
WebSocket和HTTP协议都是应用层协议，都是基于TCP传输层协议，但在设计理念、使用场景和性能特征上存在显著差异。

**HTTP：短连接vs长连接**

HTTP/1.0：短连接模式
```
客户端 → 服务器：TCP 连接建立
客户端 → 服务器：HTTP 请求
服务器 → 客户端：HTTP 响应
客户端 → 服务器：TCP 连接关闭
```
- 每个请求都需要建立新的TCP连接
- 请求完成后立即关闭连接
- 开销大，性能差

HTTP/1.1：Keep-Alive长连接
```
客户端 → 服务器：TCP 连接建立
客户端 → 服务器：HTTP 请求 1
服务器 → 客户端：HTTP 响应 1
客户端 → 服务器：HTTP 请求 2
服务器 → 客户端：HTTP 响应 2
... (可以复用连接)
客户端 → 服务器：TCP 连接关闭（超时或主动关闭）
```
- 可以在同一个TCP连接上发送多个HTTP请求
- 减少了连接建立的开销
- 但仍然受限于请求-响应模式

HTTP/2：多路复用
```
客户端 → 服务器：TCP 连接建立（TLS 握手）
客户端 → 服务器：HTTP/2 连接建立
客户端 → 服务器：流 1（请求 1）
客户端 → 服务器：流 2（请求 2）
客户端 → 服务器：流 3（请求 3）
服务器 → 客户端：流 1（响应 1）
服务器 → 客户端：流 2（响应 2）
服务器 → 客户端：流 3（响应 3）
```
- 单个TCP连接上可以并发多个请求/响应
- 解决了HTTP/1.1的队头阻塞问题
- 仍然无法实现服务器主动推送

**WebSocket：持久全双工连接**
```
客户端 → 服务器：TCP 连接建立
客户端 → 服务器：HTTP Upgrade 请求
服务器 → 客户端：HTTP 101 Switching Protocols
客户端 ↔ 服务器：WebSocket 全双工通信
... (连接保持，双向通信)
客户端/服务器：连接关闭
```
- 一次握手，长期使用
- 连接建立后，客户端和服务器都可以主动发送数据
- 连接可以保持数小时甚至数天


**各自的使用场景**
适合使用HTTP的场景
1. RESTful API：标准的CRUD操作
2. 静态资源：HTML、CSS、JS、图片等
3. 一次性数据获取：页面加载、表单提交
4. 缓存友好的数据：可以充分利用HTTP缓存机制

适合使用WebSocket的场景
1. 实时通信：聊天、通知、协作
2. 高频数据推送：监控、行情、游戏状态
3. 双向交互：在线游戏、远程控制
4. 低延迟要求：实时音视频、金融交易
